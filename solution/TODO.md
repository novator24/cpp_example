## 1. Механизмы многопоточности (Multi-threading Mechanisms)

Для ускорения обработки данных используется паттерн **MapReduce** в рамках одного узла (Shared Memory):

1. **Разделение данных (Chunking / Partitioning)**:
   - Файл, отображенный в память через `mmap`, логически делится на `N` равных частей, где `N` — количество доступных логических ядер процессора (`std::thread::hardware_concurrency()`).
   - Чтобы не разорвать слово на границе чанка, каждый поток сдвигает свое начало (`start`) вперед до первого не-буквенного символа (если предыдущий символ был буквой). Аналогично, поток читает за пределы своего `end` до конца текущего слова.

2. **Локальная агрегация (Thread-Local Storage / Map Phase)**:
   - Каждый поток создает собственную изолированную хэш-таблицу (`HashMap`).
   - Это полностью исключает необходимость использования блокировок (мьютексов, спинлоков) в самом горячем цикле парсинга слов. Потоки работают независимо друг от друга на максимальной скорости памяти.

3. **Слияние результатов (Reduce Phase)**:
   - После завершения работы всех потоков (через `std::thread::join()`), главный поток выполняет слияние (`merge`) локальных хэш-таблиц в одну глобальную.
   - Поскольку уникальных слов в естественных текстах (даже больших) относительно немного (закон Ципфа), фаза слияния занимает доли секунды.

## 2. Алгоритмы и структуры данных

Для повышения скорости работы алгоритмов по сравнению с базовой версией `0.0.1` были применены следующие подходы:

1. **Flat Hash Map с открытой адресацией (Open Addressing with Linear Probing)**:
   - Вместо стандартного `std::unordered_map`, который аллоцирует каждый узел отдельно (что приводит к фрагментации памяти и промахам кэша), используется хэш-таблица на основе единого массива (`std::vector`).
   - Это обеспечивает максимальную локальность данных (Cache Locality) и предсказание ветвлений (Branch Prediction) процессором.
   - *Ссылки/Статьи*:
     - Malte Skarupke: ["I Wrote The Fastest Hashtable"](https://probablydance.com/2017/02/26/i-wrote-the-fastest-hashtable/)
     - Google Abseil: `absl::flat_hash_map` (SwissTable дизайн).

2. **Arena Allocator (Custom Memory Pool)**:
   - Строки (слова) не сохраняются как `std::string`. Все уникальные слова пишутся подряд в один большой буфер (`std::vector<char> arena`).
   - В хэш-таблице хранится только смещение (`offset`) и длина (`length`). Отсутствуют вызовы `malloc`/`new` на каждое слово.
   - *Ссылки/Книги*:
     - Andrei Alexandrescu: "Modern C++ Design" (Глава про аллокаторы).
     - Game Engine Architecture (Jason Gregory) - использование пулов памяти для минимизации фрагментации.

3. **FNV-1a Hash (Fowler–Noll–Vo)**:
   - Очень быстрый некриптографический хэш. Вычисляется инкрементально прямо в цикле посимвольного чтения слова (Inline Hashing). Это экономит дополнительный проход по памяти слова.
   - *Ссылки*:
     - [FNV Hash Wikipedia](https://en.wikipedia.org/wiki/Fowler%E2%80%93Noll%E2%80%93Vo_hash_function)

4. **Zero-copy I/O (Memory-mapped files)**:
   - `mmap` позволяет ядру ОС напрямую отобразить файл в виртуальное адресное пространство процесса, минуя копирование из kernel space в user space буферы.
   - *Ссылки*:
     - Linux `mmap(2)` man page.

5. **Lookup Tables (Таблицы подстановок без ветвлений)**:
   - Массивы `is_alpha` и `to_lower` размером 256 байт позволяют за `O(1)` и без `if`-инструкций проверять символы и переводить их в нижний регистр. Это предотвращает сброс конвейера процессора (Pipeline Flush) из-за ошибок предсказания ветвлений (Branch Misprediction).
   - *Ссылки*:
     - Agner Fog: "Optimizing software in C++" (Глава про Branch Prediction).

6. **Parallel Sorting (Параллельная сортировка)** (Опционально, если поддерживается компилятором):
   - Использование `std::sort` с политикой выполнения `std::execution::par_unseq` (C++17) для финальной сортировки списка слов.
   - *Ссылки*:
     - [C++17 Parallel Algorithms](https://en.cppreference.com/w/cpp/algorithm/execution_policy_tag_t)
